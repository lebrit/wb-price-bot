# Архитектура

Проект рассчитан на один частный сервер и несколько доверенных Telegram-пользователей.
Основной asyncio-процесс совмещает long polling Telegram и фоновый мониторинг. Отдельный
auth-процесс обслуживает Mini App и временные Chromium; Caddy завершает TLS. Оба процесса
приложения используют одну SQLite в WAL-режиме.

```text
Telegram ── aiogram handlers ──────────────── SQLite
                  │                            ├── users / access status
                  └── monitor ────────────────┼── products / history
                         ├── public WB         ├── encrypted WB sessions
                         ├── account Chromium  └── notification outbox
                         └── MPSTATS fallback

Telegram Mini App ── Caddy HTTPS ── auth_web
                                      ├── validate initData
                                      ├── one-time auth session
                                      └── isolated Chromium → encrypted storage_state
```

## Границы модулей

- `handlers.py` / `features.py` — Telegram FSM, проверка владельца и пользовательский UX.
- `database.py` / `models.py` — транзакции SQLite, товары, история, сессии и очередь уведомлений.
- `domain.py` — деньги в копейках, basis points и чистые правила уведомлений.
- `wildberries.py` — единственная точка зависимости от изменяемой витрины WB.
- `monitor.py` — пакетные проверки, смена источника, ошибки и доставка outbox с retry/pacing.
- `charts.py` / `exports.py` — PNG-графики и переносимые CSV/JSON без секретов.
- `security.py` — нормализация browser storage state и Fernet.
- `telegram_auth.py` — проверка HMAC-подписи, `auth_date` и пользователя Mini App.
- `auth_web.py` — HTTPS-приложение, WebSocket кадров/управления и жизненный цикл Chromium.
- `cli.py` — healthcheck, backup, диагностика и операции установщика.

Публичные товары дедуплицируются между пользователями в пределах одного цикла. Товары с
аккаунтной сессией никогда не смешиваются между владельцами. Цена при смене источника
становится новой контрольной без уведомления, чтобы публичная и персональная шкалы не дали
ложный сигнал.

У товара хранится выбранный `optionId`, закреплённый `supplierId`, папка, набор тегов и JSON-массив
независимых правил. У пользователя отдельно хранятся региональный `dest`, тихие часы и время
дневной сводки. Координаты Telegram после преобразования в `dest` не сохраняются.

`auth_sessions` хранит только случайный ID, владельца, состояние и срок действия. После
проверки `initData` процесс занимает слот семафора, запускает отдельный browser process и
передаёт JPEG-кадры по WebSocket. Команды мыши/клавиатуры применяются напрямую к странице и
не журналируются. Сохранение разрешено только после появления Bearer-запроса WB. Browser
context уничтожается при успехе, отмене, disconnect или истечении 10 минут.

## SQLite

Включены `foreign_keys`, WAL и `busy_timeout`. История записывается только при изменении цены,
наличия или источника. Старые записи удаляются по retention. Backup выполняется штатным
SQLite Backup API, а не копированием активного WAL-файла.

При запуске выполняется идемпотентная миграция прежней схемы: новые столбцы добавляются через
`ALTER TABLE`, а старое одиночное правило преобразуется в JSON-массив. Повторный запуск миграции
безопасен.

Изменение состояния цены и постановка уведомления в outbox выполняются в одной транзакции.
После сбоя процесса неотправленное сообщение остаётся в SQLite и повторяется; удаление товара
или блокировка пользователя отменяет ещё не доставленные события.

Если появится потребность в нескольких экземплярах worker, следующая ступень — PostgreSQL,
lease через `FOR UPDATE SKIP LOCKED` для конкурентной обработки существующего outbox.
